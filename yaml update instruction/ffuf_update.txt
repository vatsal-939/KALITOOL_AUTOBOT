EXAMPLE USAGE:
  Fuzz file paths from wordlist.txt, match all responses but filter out those with content-size 42.
  Colored, verbose output.
    ffuf -w wordlist.txt -u https://example.org/FUZZ -mc all -fs 42 -c -v

  Fuzz Host-header, match HTTP 200 responses.
    ffuf -w hosts.txt -u https://example.org/ -H "Host: FUZZ" -mc 200

  Fuzz POST JSON data. Match all responses not containing text "error".
    ffuf -w entries.txt -u https://example.org/ -X POST -H "Content-Type: application/json" \

    ffuf -w params.txt:PARAM -w values.txt:VAL -u https://example.org/?PARAM=VAL -mr "VAL" -c

Ffuf Services:

Global Fuff Notes (Apply to ALL Services)
⦁ Global Alerts
=> At least one FUZZ/KEYWORD must exist somewhere, otherwise ffuf exits with error.
=> ffuf does not validate logical misuse of matchers/filters — invalid logic produces empty output, not errors.
=> High -t (threads) + low -timeout can cause false negatives.
=> ffuf silently ignores unsupported options in some modes (notably replay/search).

⦁ Automation Pitfalls
=> ffuf returns exit code 0 even when no matches are found
=> Always parse output file / JSON instead of relying on exit status
=> -rate and -p (delay) conflict conceptually; ffuf accepts both but behavior becomes inconsistent
